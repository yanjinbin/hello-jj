0. 先换脑子：branch 不再是本地开发单位

在 Git 里，feature branch 是开发单位。在 jj 里，开发单位是 change。 多个 change 串起来就是 stack。bookmark 只是推给 Git 远端时的名字。

你原来的习惯

branch-first

先 git checkout -b feature/foo，开发、提交、rebase main， 最后 push 这个 branch。

jj 的习惯

change-first

直接 jj new 开始一个 change。agent 可以产出一串 change。 你整理完后才 bookmark set。

1. Git 到 jj 的最小映射

你熟悉的 Git jj 里怎么想 agent coding 里的动作
commit change 是任务身份，commit id 是快照哈希 agent 每个逻辑步骤写成一个 change；内容变了 commit id 会变
git add / 暂存区 working copy 本身就是 @ 这个 commit 不用 stage；先写出来，之后用 split / squash -i 整理
feature branch 本地是匿名 change stack agent 不需要起分支名，避免命名和推送污染
git worktree jj workspace 一个 agent 一个目录，互不抢工作区
git rebase origin/main jj rebase -b @ -d main@origin review 完，把当前 stack 整体搬到最新主线后面
PR branch bookmark 最后才创建或移动 bookmark，然后 push

2. Baby steps：照着做，过一关勾一关

这套流程可以直接用于真实项目。主线名如果是 master，把命令里的 main 替换成 master。

1
准备
确认这是 jj + Git 共存仓库
jj --version
jj st
jj log --limit 5
通过标准
  • jj st 能显示 working copy。
  • jj log 能看到 @ 和父节点。
如果失败

已有 Git 仓库用 jj git init --colocate 接管。 新仓库可以直接 jj git init。

2
同步
先把主线拉到最新
jj git fetch
jj log -r 'all()' --limit 12
Git 对应动作

这一步对应 git fetch origin。你后面 rebase 的目标通常是 main 或 main@origin。

3
隔离
给 agent 新建独立 workspace
jj workspace add ../agent-task -r main -m "agent: implement task"
cd ../agent-task
jj st
通过标准

新目录存在，里面有同一份项目文件，jj st 能正常工作。

为什么这步重要

agent 可以在自己的文件系统目录里改代码、跑测试、生成文件，不会打断你当前工作区。

4
下发任务
让 agent 只产出 change，不碰远端
推荐给 agent 的任务边界

在这个 jj workspace 中完成任务。不要创建 bookmark，不要 push。 每个逻辑步骤用 jj describe -m 写清楚。需要开始下一段工作时用 jj new -m。完成后汇报 jj st、jj diff、测试结果。 

jj describe -m "feat: implement first slice"
jj new -m "test: cover first slice"
jj st
jj diff
5
观察
回到主 workspace 看 agent 产物
cd ../your-main-workspace
jj workspace list
jj log -r 'all()'
jj show <change-id>
jj diff -r <change-id>
通过标准

你能看到 agent workspace 里的 change，并且能按 change review diff。

6
整理
agent 写混了就 split
jj split -r <mixed-change>
jj split -r <mixed-change> path/to/file -m "feat: extract focused change"
什么时候用

一个 change 同时包含重构、功能、测试、格式化时，用 jj split 拆开。 这替代了 Git 里事前精细 git add -p 的压力。

7
压实
agent 修补太碎就 squash
jj squash
jj squash -i
jj squash --from <fixup-change> --into <target-change>
通过标准

最终 stack 每个 change 都有单一意图：一个功能、一个重构、一个测试补充。

8
对齐主线
执行 jj 版 rebase main
jj git fetch
jj rebase -b @ -d main@origin
jj log
选择参数
  • -b @：日常最常用，搬当前 stack。
  • -r @：只搬当前一个 change。
  • -s <root>：从某个根 change 连同后代一起搬。
Git 对应动作

这一步对应 git rebase origin/main。 jj 保留了你“主线干净”的习惯，但本地不依赖 branch。

9
发布
最后才创建 bookmark 并 push
jj bookmark set feature/agent-task -r @
jj git push --bookmark feature/agent-task --dry-run
jj git push --bookmark feature/agent-task
关键习惯

bookmark 是发布边界，不是开发边界。你 review、split、squash、rebase 完之后再挂它。

10
收尾
清理 agent workspace
jj workspace list
jj workspace forget agent-task
rm -rf ../agent-task
谨慎删除目录

workspace forget 只是让 jj 忘记这个 workspace。 rm -rf 会删除物理目录，确认里面没有要保留的临时文件再执行。

11
部署
用 GitHub Actions 直接部署到 Vercel
jj bookmark set main -r @
jj git push --bookmark main
你需要在 GitHub 配好 3 个 secrets
  • VERCEL_TOKEN：Vercel 账号 token。
  • VERCEL_ORG_ID：Vercel team/user id。
  • VERCEL_PROJECT_ID：Vercel project id。
通过标准

push 到 main 后，GitHub Actions 出现 Deploy to Vercel workflow，并完成 production deploy。

3. GitHub Actions 直接部署

本仓库已经按纯静态页面准备好：入口是 index.html， vercel.json 只保留静态站点配置，workflow 在 .github/workflows/deploy-vercel.yml。

第一次绑定 Vercel 项目

npm install -g vercel
vercel link
cat .vercel/project.json

把 orgId 和 projectId 分别填到 GitHub Secrets。 .vercel/ 不需要提交。

日常发布

jj st
jj describe -m "docs: update jj tutorial"
jj bookmark set main -r @
jj git push --bookmark main

之后每次推到 main，GitHub Actions 会执行 vercel pull、vercel build --prod、 vercel deploy --prebuilt --prod。

4. 常见坑：Git 老手要刻意避免

坑 1

一开始就建 bookmark

先让 agent 产出 change stack。bookmark 只在 push/PR 前出现。

坑 2

寻找 jj add

没有暂存区。先写，后拆。整理历史是 jj 的强项。

坑 3

用 Git 命令改状态

可以用 git status 看；尽量不要用 git reset、git checkout 改。

出错时的第一反应

不要急着 reset。先 jj op log，再 jj undo 或 jj op restore <operation-id>。
