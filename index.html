1. 为什么 agent 时代的本地工作流会变

Git 的协作生态不用替换，真正卡住 agent coding 的通常是本地状态管理： staging area、stash、临时 branch、detached HEAD、rebase in progress。 这些状态对人类有缓冲意义，但对 agent 来说只是额外上下文和额外失败点。

Git 本地流

边想边提交

你要告诉 agent 先看状态、add 哪些文件、commit 什么、切哪个分支、 stash 什么、什么时候 pull/rebase。对话里很容易混入大量版本控制仪式。

jj 本地流

先产出再整理

agent 直接在当前 change 写代码；需要下一段就 jj new。 等结果出来，再用 jj split、jj squash、 jj rebase 把历史整理好。

一句话版本

Git 继续做远端协作语言；jj 做你和 agent 在本地共同工作的操作系统。 这也是参考文章最值得吸收的判断。

2. 先理解 change：任务身份和内容快照分开

jj 的核心不是 branch，而是 change。当前工作区就是 @ 这个 working-copy commit；文件变了，它自动属于当前 change，不需要 git add。

概念 怎么理解 对 agent 有什么影响
change id 这件任务的身份。修改描述、rebase、继续编辑后通常仍指向同一件事。 review 和返工时优先说 change id，不要让 agent 猜 branch。
commit id 某一刻内容快照的哈希。文件、父节点、描述变化都会导致它变化。 适合定位某个已验证快照，不适合当长期任务名。
@ 当前正在编辑的 working-copy commit。 agent 写文件就是在改 @，不需要 add/stage。
bookmark jj 和 Git branch 的桥。推远端或开 PR 时才需要。 本地先匿名开发，发布前再命名，减少分支污染。

3. 本地变量：让命令自动对齐你的仓库

下面三个值会替换页面里的命令。默认分支可填 main 或 master；change id 可以从 jj log 复制。 commit id 字段也可以填 @- 这类本地 revset，用来自动指向当前工作区的父 change。 

jj log --limit 5
jj show {{changeId}}
jj show {{commitId}}

4. Oh My Zsh：启用 jj 插件和 alias

如果你的 shell 用 Oh My Zsh，可以直接启用内置 jj 插件。 它会生成补全、提供 prompt helper，并注册一组按子命令缩短的 alias。

启用插件

把 jj 加进 plugins

修改 ~/.zshrc 后重开终端，或执行 source ~/.zshrc。 插件本身不会替你安装 jj，先确保 jj --version 可用。 

plugins=(git jj)

source ~/.zshrc
jj --version
prompt 性能

大仓库觉得慢时再加

这个设置只影响 prompt 里插件调用 jj 的方式，会让 prompt 读取时跳过 working copy 扫描。代价是提示符可能要等你手动运行一次 jj 命令后才更新。 

zstyle :omz:plugins:jj ignore-working-copy yes

和 jj 自带 alias 分清楚

Oh My Zsh alias 是 shell 缩写，例如 jjst；jj config set --user aliases.l ... 是 jj 自己的子命令 alias，例如 jj l。两者可以同时存在。

5. 实战流程：把 agent 产物变成可推送历史

这不是 jj 命令大全，而是一条可以每天重复的 workflow。每一步都刻意减少 agent 对 Git 隐性状态的依赖。

1
接管仓库
让 jj 和现有 Git 仓库共存
brew install jj
jj git init --colocate
jj st
jj log --limit 5
通过标准

jj st 能显示 working copy，jj log 能看到 @。远端仍然是 GitHub/Git，不需要迁移代码托管。

2
开始任务
给当前 change 写清楚意图
jj describe -m "feat: implement first agent slice"
jj st
jj diff
给 agent 的边界

只要求它完成业务目标、跑验证、汇报 diff。不要一开始就让它建 branch、 add、commit、push。

3
继续下一段
用 jj new 代替提交仪式
jj new -m "test: cover first agent slice"
jj st
jj log --limit 8
心智模型

jj new 等于“前一段工作定格，开一个新的空 change 继续”。 它不是 Git 那套 add + commit + checkout 的组合仪式。

4
中途打断
临时修 bug 不用 stash
jj new {{defaultBranch}} -m "fix: handle urgent bug"
jj st
# 修完后回到之前的 change
jj edit {{changeId}}
为什么比 stash 简单

你不是把当前工作临时藏起来，而是从主线开一个新的 change。 原来的工作仍在历史图里，有名字、有位置，可随时回去。

5
拆分产物
agent 写混了，用 split 整理
jj split -r {{changeId}}
jj split -r {{changeId}} path/to/file -m "feat: extract focused change"
jj log --limit 10
关键收益

不需要先 reset 再反复 add -p。拆错了可以继续 edit/split/squash， 后续 change 会自动跟着调整。

6
预排计划
先建 change 骨架，再让 agent 填内容
jj new -m "refactor: extract module boundary"
jj new -m "feat: implement behavior behind boundary"
jj new -m "test: cover behavior and regression"
jj log --limit 10
适合复杂任务

把 spec 拆成一串有描述的空 change。agent 逐个 jj edit 填实现，change 描述本身就能变成验收标准。

7
对齐主线
fetch 和 rebase 分成两步
jj git fetch
jj rebase -b @ -d {{defaultBranch}}@origin
jj log --limit 12
Git 对应动作

这一步相当于 git fetch 后再明确决定 怎么整合到 {{defaultBranch}}@origin。

8
发布
整理完再挂 bookmark 并 push
jj bookmark set feature/agent-task -r {{changeId}}
jj git push --bookmark feature/agent-task --dry-run
jj git push --bookmark feature/agent-task
不要过早命名

bookmark 是发布边界，不是开发边界。push 前先 review、split、 squash、rebase，最后才把目标 change 暴露给 Git 世界。

6. 六个 agent 场景的 jj 解法

下面是参考文章的核心场景，用更偏操作手册的方式压缩成可执行判断。

开始下一项

不要先建 feature branch

让 agent jj new -m "..."，直接开始下一段业务工作。

中途插队

不要 stash

从默认分支开一个新 change，修完再 jj edit 回来。

产物太乱

先收下，再 split

让 agent 先跑通，再按功能、测试、文档拆 change。

任务复杂

先做 change 骨架

把计划写进多个空 change 的描述，让 agent 逐个填实现。

多 agent

一个 workspace 一个 agent

用 jj workspace add 分目录并行，完成后按 change 合并。

搞砸了

先 jj undo

操作级回滚比猜 reset/revert/checkout 的副作用更可靠。

7. 可以直接给 agent 的指令

把版本控制要求写得短而明确：让 agent 用 jj 工作，但不要让它提前发布。 

这个仓库本地使用 jj 管理版本控制。
完成任务时：
1. 在当前 change 中实现，不要创建 bookmark，不要 push。
2. 每个逻辑阶段用 jj describe -m 或 jj new -m 写清楚描述。
3. 如果一个 change 混入多种意图，用 jj split 拆分。
4. 出错先用 jj st、jj log、jj op log 说明状态，不要直接 git reset。
5. 完成后汇报 jj st、jj diff、测试命令和结果。

8. 最小命令速查

要做什么 jj 命令 Git 直觉
看状态和历史 jj st / jj log / jj diff git status + git log
描述当前工作 jj describe -m "..." 写 commit message，但之后还能轻松改
开始下一段 jj new -m "..." 定格当前工作并继续，不需要 add
回旧 change 修改 jj edit {{changeId}} 类似 checkout 某个提交，但不会进入 detached HEAD 心智负担
拆分 change jj split -r {{changeId}} 替代 reset + add -p + 多次 commit
撤销上一步 jj 操作 jj undo 比 reflog + reset 更直接
同步远端 jj git fetch git fetch
对齐主线 jj rebase -b @ -d {{defaultBranch}}@origin git rebase origin/{{defaultBranch}}
发布成 Git branch jj bookmark set feature/name -r @ 后 jj git push 最后才给远端协作命名
